01

設計原則(七條,全部已實裝)

1. Adapter Interface Pattern — 每個外部資料源一個 adapter,實作統一介面 BaseAdapter.fetch(**kwargs) → list[RawLead]。任一資料源漲價/封鎖/關閉,換 adapter 不動核心。寄送後端同理:eml / gmail 可插拔。

2. Human-in-the-loop 閘門 — 系統只產草稿,人工核准後才進入自動寄送。覆核頁一次審整串三輪(皆可改稿),按一次核准 → 排程 → 自動寄 → 回信自動煞車。AI 幻覺永遠過不了人眼這道門,但人只按一次。

3. 模型分級用工 — 便宜快的做清洗抽取,中階做生成判斷,最貴的只做審稿把關(見 02 章映射表)。

4. 單機優先 — SQLite 單檔 + 本地執行;FastAPI 輕量 Web UI。不引入 Postgres/佇列服務/雲端。

5. 戰略防線內建 — 戰略規則寫進程式:T3 不主動觸達、T0 Rep 獨立通道、同收件人至多 3 封、回信即停。違反戰略的操作被系統攔截,不依賴人的記性。

6. 每個判定可解釋 — 評分附 rationale、去重逐筆印原因、佇列每筆狀態/錯誤可查、排程器逐輪日誌。人可以審計每一個機器決定。

7. 確定性歸程式,判斷歸 LLM(v3 明文化) — 排程數學、限流、狀態機、合規 footer 等寫得成 if/else 的規則一律用程式實作並測試釘死;LLM 只出現在四個判斷點(背景研究/契合評分/寫信/審稿)。整個 L6 寄送層零 LLM 呼叫。

9.1 入佇列(sequence.enqueue_for_lead)— 核准的那一下

  • 守門:缺 email / 已退訂 / 該 lead 已有 ready 佇列 → 拒絕(ValueError 帶人話)
  • 排程:seq1 = buyer 當地(州別→IANA 時區,查無用 FALLBACK_TIMEZONE=美東)最近的寄信時段起點(工作日 09:30–16:30,跳週末與美國聯邦假日 2026–27);同日已有排程 → 錯開 random(60–90) 分鐘,溢出時段順延次工作日;seq2/3 = seq1 日 +4/+6 工作日的時段起點
  • 展中特例:stage=met_at_show 只排一封、排「現在」(24 小時跟進 KPI 不等隔天時段)
  • 每封 body 自動附 footer(9.4);寫入 email_queue(status=ready);清空 pending 草稿並記互動

9.2 佇列狀態機

ready ──派送成功(拿到 message_id)──▶ sent
  │──回信/退訂/歸檔/超上限──▶ cancelled(永久;UI 亦可手動取消)
  │──後端失敗──▶ failed(UI 可一鍵重排回 ready)
  └──seq1 未寄出時 seq2/3 到期──▶ skip(暫緩,保留 ready)——不誤殺跟進

9.3 排程器(dispatcher.run_once)— 防風控節奏的心臟

規則實作防什麼
一次只寄 1 封到期多封仍只取最早那筆,其餘留下一輪批次爆寄=最典型的機器訊號
暖機每日上限起算日=env 或第一封實際寄出日;前 2 週 2 封/天,之後 5 封/天(env 可調)新帳號突然放量
全域 interval 護欄距上一封寄出 < INTERVAL_MIN(60 分)→ 本輪不寄多 lead 的 seq2/3 撞在同一時刻(+4 工作日的 09:30)被 60 秒輪詢連發
寄前重查寄出前一刻重查退訂名單與 lead 階段;該取消就取消、該暫緩就暫緩(9.2)核准後狀態變化(對方回信/退訂)
warm 通道測試信(test=True)與展中 follow-up(met_at_show)豁免每日上限與 interval,優先出門回覆認識的人不是 cold outreach,不該被限流卡死
不謊報拿到後端 message_id 才標 sent + 更新 lead(seq1→contacted、排 next_action_due);失敗標 failed 進日誌「宣稱已寄出」與事實脫鉤
bounce 防線同網域累積 3 次失敗 → 整網域自動加入退訂名單高退信率毀寄件信譽
thread 接續seq2/3 取 seq1 的 thread_ref 傳 threadId(gmail 模式)對話串體驗,不是三封新信轟炸

9.4 合規 footer 與退訂機制(footer.py + List-Unsubscribe)

CAN-SPAM 兩要件:寄件人實體地址(Profile sender.address)+ 可運作的退出機制——本系統提供兩路:(1)footer 文字指示 reply UNSUBSCRIBE(10 個工作日內處理);(2)每封信帶 List-Unsubscribe: <mailto:寄件人?subject=UNSUBSCRIBE> 標頭——Gmail 等收件端據此在信件頂部顯示原生「取消訂閱」按鈕(mailto 變體免 web endpoint,對寄達率為正面訊號)。按鈕或回信觸發的退訂信由回覆偵測自動處理(9.5)。ENABLE_COMPLIANCE_FOOTER=0 時退為純簽名(標頭仍附)。address 為佔位字時:footer 印 ⚠ 提示、UI 紅色警告、gmail 自動寄送被硬性擋下。退訂名單(email 或整網域)寄前必查;「清空全部資料」保留退訂表。

9.5 寄送後端(可插拔,SENDING_BACKEND)

後端機制用途
eml(預設)到期時輸出標準 .eml(To/Subject/X-Unsent)至 outbox/,message_id=檔名乾跑:排程/限流/狀態機全部真實運作,「送出」由人;網域暖機完成前的安全模式
gmailREST + OAuth refresh token(httpx,零新依賴;scope 僅 gmail.send + gmail.readonly);access token 快取;送 users/me/messages/send(可帶 threadId)全自動寄送;回覆偵測:每輪掃 sent thread,出現非我方訊息 → apply_track(replied) → 取消剩餘跟進;回信含 UNSUBSCRIBE/退訂關鍵字 → 自動加入退訂名單 + 歸檔(對應 List-Unsubscribe 按鈕與 footer 指示)

9.6 背景排程器(webui/scheduler.py)

  • serve 啟動時開 daemon thread,每 60 秒 run_once(+ gmail 模式順帶 check_replies);寄送節奏由 scheduled_at 錯開與限流控制,輪詢頻率與寄量解耦
  • 暫停/恢復:marker 檔(重啟記得);手動「立刻跑一輪」不受暫停影響
  • ring buffer 日誌(300 行)供 /outbox 即時顯示;連續空轉去重不洗版
  • 已知單點:排程器生命週期 = serve 行程(技術債 #1,見 17 章)

9.7 🧪 測試信(v3 新增)

/outbox 一鍵寄測試信給指定信箱:走完整鏈路(佇列→排程器→後端→footer)但 test=True ——不佔暖機額度、不啟動暖機時鐘、豁免 interval、不動 lead 階段、pipeline 跳過測試 lead。2026-07-10 端對端實測:自動寄達(落點垃圾郵件——成因與對策見《上線作戰手冊》,屬帳號/網域信譽問題,非系統缺陷)。
