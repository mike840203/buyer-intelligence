01

設計原則(六條,全部已實裝)

1. Adapter Interface Pattern — 每個外部資料源一個 adapter,實作統一介面 BaseAdapter.fetch(**kwargs) → list[RawLead]。任一資料源漲價/封鎖/關閉,換 adapter 不動核心。

2. Human-in-the-loop — 系統只產草稿,絕不自動寄信。核准 → .eml 輸出 → mailto 開啟郵件軟體 → 寄送鍵由人按。

3. 模型分級用工 — 便宜快的做清洗抽取,中階做生成判斷,最貴的只做審稿把關(見 02 章映射表)。

4. 單機優先 — SQLite 單檔 + 本地執行;展中以 FastAPI 輕量 Web UI 供手機使用。不引入 Postgres/佇列/雲端。

5. 戰略防線內建(v2 新增)— 戰略規則寫進程式:T3 不主動觸達、T0 Rep 走獨立信件通道。違反戰略的操作被系統攔截,不依賴人的記性。

6. 每個判定可解釋(v2 新增)— 評分附 rationale、去重逐筆印原因與保留者職稱、落選聯絡人轉備援不丟棄。人可以審計每一個機器決定。

L2 清洗豐富層 — 完整規格

5.1 去重演算法(dedupe)

同一家公司的判定(任一成立即同公司):

  • 公司 email 網域相同——但免費信箱不算證據:gmail.com / yahoo.com / outlook.com / hotmail.com / icloud.com / aol.com / live.com / msn.com / me.com / protonmail.com / proton.me 一律排除(兩家小店都用 gmail ≠ 同一家)
  • 公司名正規化後模糊比對 ≥ 90(rapidfuzz token_sort_ratio):正規化 = 轉小寫 → 剝除法律後綴(inc/llc/ltd/corp/corporation/co/company/group/holdings)→ 去標點

同公司多聯絡人時保留誰(_contact_priority,降冪排序):

優先級條件(職稱關鍵字,不分大小寫)邏輯
4(最高)買手詞(buyer/category/purchas/merchandis)且品類詞(hardgood/housewares/home/kitchen/coffee/seasonal/decor/tabletop/gourmet)品類對口買手就是門
3owner / founder / ceo / president小店店主即採購;公司大到有品類買手時買手勝出
2泛買手詞在採購鏈上
1有職稱同級以欄位完整度(email/姓名/職稱/網站計數)決勝
0無職稱

全部聯絡人保留(不丟任何人):同公司其他聯絡人結構化存入 alt_contacts(姓名/職稱/email 完整保留);UI 詳情頁列出全部聯絡人,「設為主收件人」一鍵切換(原主收件人退為備選,換人自動重置 email 驗證旗標)——優先序演算法只決定預設值,收件人最終由人判斷。verbose 模式逐筆印「✂ 併入備選 …(同公司已保留 ○○○(職稱)——判定依據:…)」。

5.2 email 驗證(verify_email)

GET api.hunter.io/v2/email-verifier;狀態 valid / accept_all / webmail 判定有效。未設金鑰或網路錯誤 → 標 email_verified=False,不阻擋流程(覆核時人工注意)。

執行時機(v2 修正):在 pipeline L2 背景節點執行(enrich_lead 開頭),不在匯入時——同步逐筆打 API 曾使數十筆匯入卡住 UI 逾一分鐘;背景長工作多 1 秒/筆無感。例外:UI 手動修改信箱或切換主收件人時當場驗證該筆(~1 秒)。順序保證:驗證必發生於覆核寄信之前(pipeline → review)。

5.3 背景豐富(兩段式)

步驟模型Prompt 規格輸出
research_companySonnet + WebSearch(≤5 次)以公司名+網站+城市查:(1)門市數或純電商 (2)通路類型(咖啡器材電商/烘豆連鎖/廚房專賣/一般零售/rep group)(3)是否已賣競品(Fellow Atmos、Planetary Design、Airscape、OXO POP)(4)營收規模;≤200 字自由文字摘要

回填 Lead:store_count / sells_competitors / enrichment_notes。州別入庫時正規化為縮寫(50 州全名對照表,Apollo「Illinois」與 Places「IL」統一);地區僅供評分權重內部使用:WA,OR→PNW;TX→TX;CA→CA;NY,NJ,CT→NY;IL,WI,MI,MN,OH,IN→MIDWEST;其他→OTHER。UI 與篩選一律用州名。

6.2 一般評分:總分 = 0.40×通路契合 + 0.25×規模 + 0.20×地區 + 0.15×決策權

設計原則:能用規則算的用規則(透明、零成本、可複查),需要判斷的才交給 AI。四維中三維(規模/地區/決策權規則部分)是明文死規則,分數表如下、權重全在 config.py 可改;只有通路契合 40% 與決策權的一半是 LLM 判斷。整筆結果不是黑箱——每筆寫入白話 score_rationale(各維得分 + LLM 加減分理由),資料庫存、Web UI 名單詳情頁「評分依據」欄直接顯示。

維度(權重)計算方式分數表
通路契合(40%)LLM(Sonnet 結構化):輸入公司/職稱/tier/背景摘要,輸出 FitJudgment{channel_fit_score 0-100, authority_score 0-100, rationale};prompt 載明優先序「咖啡器材 > 廚房專賣 > 一般零售」與「已賣競品加分」;解析失敗保底 = 雙 50 分中性0–100
規模適配(25%)規則,store_count5–100 家=100;<5=60;101–500=40;>500=15;未知=50
地區(20%)規則,regionPNW=100、TX=100、CA=80、NY=80、MIDWEST=60、OTHER=30
決策權(15%)(規則 + LLM authority_score)÷2;規則 = 職稱關鍵字owner/founder=100、ceo/president=95、buyer/category manager=90、merchandis/purchasing=85、director=70、manager=55、其他/無=30

6.3 分級與條件分流

≥70 → A(進 L4,信件逐封人工覆核);50–69 → B(進 L4);<50 → C(stage=archived,不觸達,不再花任何成本)。每筆寫入完整 score_rationale(各維分數+LLM 理由)。

實測校準:首批名單 P1 地區公司全數 A(73–78 分)、NY 71 分 A、KY(OTHER)62.7 B、29,000 人飲料集團被 T3/低契合攔下——與戰略地區優先序一致,權重免調。

透明度實例(Intelligentsia Coffee):通路契合 82(40%)、規模 100(14 家店在甜蜜點,25%)、地區 60(IL,20%)、決策權 52(15%)→ 總分 77.7 = A。LLM 加分理由「官網已賣 Fellow Atmos,品類有貨架」;扣分理由「但聯絡人是 Green Coffee Buyer——採購生豆的上游,與零售端器材選品分屬不同部門,對配件上架影響力有限」。此即 AI 的價值:規則看職稱都有 Buyer 字樣分不出,AI 讀懂咖啡業部門分工、把決策權從高分壓到 52。整段理由 UI 可見,人可審計、可覆核時推翻。

L4 觸達引擎 — 完整規格

7.1 信件生成(draft_email,Sonnet)— 硬性要求(prompt 載明)

  • 全英文三重防線:(1)claude -p 附加 --append-system-prompt 語言覆寫(壓過使用者全域 CLAUDE.md 的「一律中文回覆」指令,此為實測發現的 CJK 混入根因);(2)生成後 CJK 正則偵測(含日文假名),命中即帶警告重生成一次;(3)Opus 審稿把「含任何非英文字元」列為 instant fail
  • 純文字、<150 字、第一行 Subject: …、無列點、無 emoji、無 hype 詞
  • 一句話價值主張(固定敘事:台灣設計製造手搖真空棘輪、免電池免泵、內部保鮮實測)
  • 為何找上「這一家」的具體理由——只准使用 L2 查到的事實,嚴禁捏造;查無資料時保持誠實泛寫
  • TIHS 攤位資訊(env:TIHS_BOOTH)+ Calendly 連結(env:CALENDLY_URL)
  • T0_rep 改走代理合作敘事(line representation、佣金制、美國零售關係),不談批發採購
  • 重寫輪帶入審稿者 rewrite_hints

7.2 審稿(critique_email,Opus)— fail 條件(任一成立即 revise)

角色設定:「你是 {company} 的 buyer,一天收幾十封供應商冷信,見罐頭即刪」。輸出 CritiqueResult{verdict: pass|revise, issues[], rewrite_hints}。fail 條件:超過 150 字/像模板/「為何是你們」理由空泛或捏造/價值主張不清/語氣推銷味或浮誇/缺攤位或預約連結。解析失敗保底 = pass 並標注「請人工加強檢查」(人是最後防線)。

7.3 迴圈與覆核

draft ⇄ critique 最多 3 輪,超限強制進覆核。過稿 → pending_draft 掛上 lead 入庫。review 指令逐封顯示草稿+評分依據:y=核准(輸出 outbox/<id>_<公司>.eml,含 To/Subject/X-Unsent 標頭;stage→contacted;排 5 天後跟進;無 email 者警示)/n=退回清稿/s=跳過。

7.4 寄送(send)

send <id> → 取最後一封核准信 → mailto: URL(收件人/主旨/內文 URL-encode)→ 開啟預設郵件軟體撰寫視窗。寄送鍵永遠由人按。

對所有 stage=met_at_show 且無待審草稿者,依當日會談紀錄生成 same-day follow-up(Sonnet,<120 字、必須引用會談具體內容、載明下一步)→ 進覆核佇列 → review 掃過寄出。支撐 KPI:24 小時跟進率 100%。

class Lead(BaseModel):
    id: int | None                       # SQLite rowid
    # 身分
    company: str;  contact_name/title/email/website/city/state: str | None
    email_verified: bool = False;  source: str   # apollo/places/iha/manual/ocr
    # 分類
    tier: "T0_rep|T1_coffee|T2_kitchen|T3_mass"
    region: "PNW|TX|CA|NY|MIDWEST|OTHER"
    store_count: int | None;  sells_competitors: bool | None
    enrichment_notes: str | None            # 背景摘要 + 備援聯絡人
    # 評分
    score: float | None;  grade: "A|B|C" | None;  score_rationale: str | None
    # Pipeline
    stage: "new→contacted→meeting_booked→met_at_show→followed_up→sample_sent→quoting→po_received | archived"
    interactions: list[Interaction]         # email_draft/email_sent/meeting_note/follow_up/other,全文入庫
    next_action_due: date | None            # 逾期警示依據
    pending_draft: str | None               # 人工覆核佇列

# 中間模型:RawLead(L1 統一輸出)、CardContact(OCR)、EnrichmentFacts(L2)、
#           FitJudgment(L3)、CritiqueResult(L4)——均為 LLM 結構化輸出 schema

儲存:SQLite 單檔 data/leads.db;表 leads(id, company, email, tier, region, grade, stage, data JSON),常用查詢欄位建索引,完整 Lead 以 JSON 存 data 欄。查詢介面:save_lead / get_lead / list_leads(stage, grade) / find_by_company_or_email / overdue_leads。

已知限制與後續路線

  • 寄信基礎設施(系統範圍外,成敗前提):專用網域 + SPF/DKIM/DMARC + 4–6 週暖機 + CAN-SPAM(實體地址/退訂)須於 outreach 前完成——已排入戰略時間軸 2026-08。
  • 回覆偵測是手動的:誰回信靠人發現後 track replied;放量後接 Gmail API 自動偵測(規劃中,非必要)。
  • B 級批次覆核未實作:review 逐封;量產期若成瓶頸再加批次模式。
  • 評分權重為先驗設定:展後以實際回覆率/會議轉化率校準 config.SCORE_WEIGHTS。
  • 會談紀錄僅文字:v1 規劃的語音輸入未做(手機鍵盤語音輸入可替代)。
  • Web UI 無登入機制:設計為本機/內網單人使用;若需對外開放須加認證層。
  • Web UI 匯入僅支援 CSV:Apollo API(付費後)與 Places 掃描目前走 CLI ingest;需要時再加進 UI。
